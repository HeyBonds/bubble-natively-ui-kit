Bonds Tech Roadmap

From Prototype to Platform

3-phase migration strategy — Bubble + Natively → Fully Native

Where We Are Today

The app currently runs as a Bubble-native app wrapped with Natively.

All UI — rendered by Bubble's built-in page system (Bubble elements, responsive engine, workflows)

Natively — wraps it in a native shell for App Store + bridge access to device capabilities

Result — it works, but every UI change means fighting Bubble's layout engine, and the experience feels like a web app — because it is one

Platform Limitations (Today & Phase 1)

Inherent to WebView + Bubble. Persist through Phase 1 — only Phase 2 and 3 resolve them.

  • WebView performance ceiling — browser engine inside the app. Can't match native rendering for scroll-heavy lists, gesture-driven animations, or complex transitions.
  • Bubble vendor lock-in — business logic in visual workflows. Can't version-control, can't unit test, can't migrate without rewriting.
  • Bridge overhead — every native call round-trips: JS → Natively → OS API → Natively → JS. Adds latency, limits architecture.
  • No push / offline / background — WebView can't register for push tokens, run background tasks, or do offline-first sync. Native-only capabilities.
  • Scale costs — Bubble pricing scales per user/workflow. No control over infrastructure.
We validated with Bubble. Now we need infrastructure to scale.

Why React? The Vibe Coding Advantage

  • Largest ecosystem in frontend — most libraries, most Stack Overflow answers, most open-source code. AI models are trained on massive amounts of React — they generate higher-quality React code than any other framework.
  • Declarative, component-based — self-contained units map perfectly to AI code generation. Describe what a component should do, get a complete working component.
  • JSX is readable by humans and machines — markup + logic in one file. No template language, no separate files to sync. AI tools understand context and generate correct output.
  • React → React Native is the same paradigm — hooks, state management, component patterns all transfer. AI tools that generate good React code also generate good React Native code.
Phase 1 investment in React accelerates Phase 2. AI-assisted development works across both.

Phase 1 — What's Ahead

Phase 1 is far from done. The app is large, and migrating each screen involves significant work.

  • Screen-by-screen migration — every Bubble page rebuilt as a React component with new UI, styling, and interaction logic
  • Workflow restructuring — moving to single-page React components changes how Bubble workflows operate. Frontend workflows shift from driving UI to responding to JS2B events. This touches every screen.
  • JS2B integration per feature — each feature needs its own Toolbox element with Multiple Outputs, action routing, and wired Bubble workflows
  • Testing across devices — each migrated screen needs validation to match or exceed the Bubble original
Key change: Bubble workflows shift from driving UI directly → responding to JS2B actions from React.
Mostly frontend workflow changes, heavily reliant on JS-to-Bubble bridge.

Phase 1 — Specific Challenges

Beyond the platform limitations, Phase 1 introduces its own challenges:

  • Two UI paradigms during migration — some screens are Bubble pages, others are React components. Users see inconsistency; team maintains two rendering approaches at once.
  • Workflow restructuring overhead — every screen migration requires rearchitecting its Bubble workflows. Frontend workflows shift from driving UI to responding to JS2B events — non-trivial, touches both sides.
  • JS2B complexity scales with features — each screen adds another Toolbox element, more action routing, more Bubble workflows. Coordination overhead grows with app surface area.
  • Debugging across the bridge — bugs can live in React, in Bubble workflows, or in JS2B communication between them. Tracing issues across this boundary is harder than debugging either side alone.

  • True native performance — 60fps animations, native scroll & gestures
  • Push notifications — FCM / APNs, fully controlled

Bubble's role: headless backend via Data API + Workflow API
No Bubble-side changes needed → zero backend migration risk

Phase 3 (FUTURE)

React Native + Real Backend

Replace Bubble backend with real infrastructure. Frontend stays unchanged from Phase 2.

Value

  • Full control over logic & data
  • Version-controlled, testable code
  • No per-user Bubble costs
  • Optimized queries & caching
  • Any integration, any pipeline

When to Trigger

  • Bubble costs > self-hosted costs
  • Logic exceeds Bubble workflows
  • Need WebSockets, ML, etc.
  • Performance requirements grow
Phase 3 is optional until scale demands it.
